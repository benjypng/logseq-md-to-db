# CLAUDE.md — guide for modifying this importer with an LLM

This project is a single-file importer (`import.py`, lives in the root of a Logseq 1.0 Markdown graph) that rebuilds the graph in a new Logseq DB graph by driving the `logseq` CLI. Users are expected to adapt it to their own graphs. This file gives you (the LLM) the architecture, the verified CLI behavior, and the recipes to make changes safely.

## Commands

```bash
python3 -m pytest test_import.py -v          # full test suite; must stay green
python3 import.py --dry-run --no-convert-tags --import-properties   # parse + report only, no CLI calls
python3 import.py --no-dry-run --graph <name> ...                   # real import into a NEW graph
```

`import-report.txt` (next to the script) is regenerated by both modes and is gitignored.

## Hard constraints

- `import.py` stays a **single file**, **standard library only**, **Python 3.9 compatible** (no `X | None` annotations; `argparse.BooleanOptionalAction` is the floor).
- The codebase is intentionally comment-free; it relies on self-documenting names. Match that style.
- The importer must **never write into an existing graph**: `Importer.create_graph` checks `graph list` before `graph create` (which is silently create-or-open — see CLI facts below). Do not weaken this.
- Every lossy or skipped conversion must leave a trace: an `Issue` (with file and line) or a counter that `write_report` prints. Silent drops are bugs.
- Error policy in the real run: print red via `print_error`, record in `self.failures`, skip, continue. Exit code 1 if failures, 2 for aborts.

## Architecture (data flow)

```
scan_graph(root, opts)                      one parse pass, shared by dry run and import
  page_name_from_filename / journal_page_name
  parse_page_text -> Page                   preamble props, block splitting, tree building
    split_page_text -> [RawBlock]           bullets + continuation lines
    process_block -> Block                  task markers, priorities, SCHEDULED/DEADLINE,
                                            id:: uuid consumption, LOGBOOK drop,
                                            #+BEGIN_QUERY capture, properties via handle_property
    build_tree                              depth stack -> children
    finalize_block                          inline #tags, {{embed}} link detection,
                                            scan_content_features (refs/assets/embeds/queries),
                                            empty-block pruning, counters
validate(model)                             unresolved ((uuid)) refs, page-name collisions
write_report(model, path, mode, ...)        counts + issues grouped by kind + failures + notes

Importer(model, graph_name, cli).run()      real import only
  create_graph                              existence guard, then graph create
  create_tags -> {name: db_id}              upsert tag per name (must exist before use)
  create_properties -> {name: ident}        upsert property, then list property --with-type
  import_pages                              TWO loops: (1) upsert every page, capture
                                            page_ids; (2) per page, one batched
                                            upsert block --blocks-file with the whole tree
  task_followups                            upsert task --uuid for SCHEDULED/DEADLINE
  query_followups                           patch each #Query's value entity (see below)
  link_followups                            block/link for embeds unresolvable at creation
```

Key dataclasses: `Options` (convert_tags, import_properties), `Block` (content, uuid, tags, properties, status, priority, scheduled, deadline, query, link_page, link_uuid, children, line), `Page`, `GraphModel` (pages, issues, counters, uuids, block_refs, tag_names, prop_names), `Issue(kind, file, line, detail)`, `ParseContext(file, opts, model)`.

`LogseqCli.run(*args, timeout_ms=None)` shells out to `logseq <args> --graph <g> --timeout-ms <ms> --output json`, returns the parsed `data`, raises `CliError` otherwise. Tests never invoke the real CLI — they use the `FakeCli` recording double in `test_import.py`.

## Verified logseq-cli facts (each was tested against a live DB graph — trust these)

**Batched block creation** (`upsert block --blocks-file <edn>` — one call per page, create-only):
- `:block/uuid #uuid "..."` is preserved — this is how `((uuid))` refs survive.
- `:block/children [...]` nests arbitrarily deep; returns db/ids parents-first.
- `:block/tags [<int-db-id> :logseq.class/Task]` — user tags by db id (captured from `upsert tag`), built-in classes by ident keyword.
- `:logseq.property/status :logseq.property/status.todo` and `:logseq.property/priority :logseq.property/priority.high` work by ident. Status idents: `.backlog .todo .doing .in-review .done .canceled`; priorities: `.low .medium .high .urgent`; task class: `:logseq.class/Task`; query class: `:logseq.class/Query`; code class: `:logseq.class/Code-block`.
- `:user.property/<name>-<suffix> "value"` sets user properties; a value like `"[[Page]]"` resolves to a page reference.
- `:block/link <int-db-id>` or `:block/link [:block/uuid #uuid "..."]` creates a node embed with an empty title — but the lookup-ref target must already exist in the DB at transact time.
- `[[Page]]` inside `:block/title` auto-creates the referenced page. Inline `#tag` text and `((uuid))` text are stored verbatim (no ref datoms).

**Tags and properties:**
- `--update-tags` fails with `tag-not-found` unless the tag exists — always `upsert tag --name <n>` first (returns the db id).
- `upsert property --name foo` generates a random-suffix ident (`user.property/foo-Xxxxxxxx`); recover it via `list property --with-type` (JSON `db/ident` has **no leading colon**).
- `--update-properties` and `--update-tags` **do not combine in one call** — the tags silently don't apply. Always two separate calls.
- `--update-properties` accepts schema attributes too (e.g. `{:block/link 4483}`) but only literal values — a lookup ref here fails with "value should be a string".

**Advanced queries** (the `#+BEGIN_QUERY` → `#Query` conversion):
- The query text goes in as a plain string property `:logseq.property/query "..."` (no backtick fences, string must start directly at `{` — lstrip the body).
- The CLI auto-creates a value entity for it. To match the GUI's advanced-query editor, that value entity must then be patched: `debug pull --uuid <block>` → `entity."logseq.property/query"."db/id"` → `upsert block --id <valId> --update-properties '{:logseq.property.node/display-type :code :logseq.property.code/lang "clojure"}'` → separate `--update-tags '["Code"]'` call.
- Nesting these attributes in the blocks EDN is rejected (`"Query" should be a text block`).

**Graphs and safety:**
- `graph create` is **create-or-open**: it NEVER fails when the graph exists. The existence guard must check `graph list` (JSON `data.graphs` is a list of name strings) first.
- `upsert page --page "2026-07-17"` creates a real journal page (`:block/journal-day`). Page upserts are idempotent and return the db id — that's how `page_ids` gets built.
- `remove` is a soft delete (recycle bin); `debug pull` is ground truth when a query surprises you.
- Setting `:block/link` on an *existing* block works (int id only) but the block's title behavior after that varies by CLI build — the creation-time path is the one that reliably matches the GUI shape.

## Recipe: adding a new conversion

Follow the pattern set by the query and embed features (read their code first — they are worked examples of this recipe):

1. **Find the target shape empirically.** Have the user create the desired result in the Logseq GUI on a test graph, then `logseq debug pull --id/--uuid ... --output edn` it. Diff against what a naive CLI insert produces. The GUI-created entity is the spec.
2. **Verify the write path on a scratch graph** named `ls-import-<something>-qa`: create it, test the exact upsert calls, `debug pull` to confirm byte-level match, then `logseq graph remove` it. **Never touch the user's real graphs.**
3. **Parser**: add a field to `Block`, populate it in `process_block` (drawer/marker-level syntax) or `finalize_block` (whole-content patterns), keeping the original text as fallback where sensible. Record an `Issue` and/or counter for every occurrence and every edge case (unclosed, duplicate, unresolvable).
4. **Emission**: extend `block_edn` for creation-time attributes; add an `Importer.<x>_followups()` pass (modeled on `query_followups`/`link_followups`) when the write needs the rest of the graph to exist first. Skip pages in `self.failed_pages`.
5. **Report**: add `ISSUE_EXPLANATIONS` entries for any new issue kind (only kinds that actually fire) and new counters to `write_report`.
6. **Tests first** (TDD): parser tests on the new field, emission tests on the EDN string, importer tests with `FakeCli` asserting the exact call sequence. Then a live e2e on a scratch graph, then the real dry run over the graph to check the counts.

## Testing conventions

- `test_import.py` loads the script via importlib as `ls` (the filename `import.py` is not importable). Helpers: `make_ctx(...)`, `make_model_for_import()`, `FakeCli` (records `.calls`, `fail_on` substring triggers `CliError`; extend its canned responses when a new CLI call shape is added).
- The dry run is pure parsing — safe to run against a real graph anytime, no CLI needed.
- Mutation-check new failure-path tests: temporarily break the code and confirm the test fails. Several vacuous tests were caught this way.
